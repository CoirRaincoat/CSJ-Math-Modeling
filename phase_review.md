## 二、输出结果数据合理性评估

### 2.1 May 2025 预测数据 (`p2_may2025_predictions.csv`)

**数据概览（22个工作日）**:

| 指标 | 值 | 评估 |
|------|-----|------|
| 日均订单预测 | 258人 | ✅ 与历史均值274人接近 |
| 日均销售额预测 | 2,830元 | ✅ 与历史3,033元接近 |
| 预测模式 | 5值完全重复（278/238/273/273/247 每5工作日循环） | ❌ **不合理** |

**核心问题：预测值呈现精确的5天周期性重复**

```
2025-05-01 Thu: 278 → 2025-05-08 Thu: 278 → 2025-05-15 Thu: 278 → ...
2025-05-02 Fri: 238 → 2025-05-09 Fri: 238 → 2025-05-16 Fri: 238 → ...
2025-05-05 Mon: 273 → 2025-05-12 Mon: 273 → 2025-05-19 Mon: 273 → ...
```

**根因**：`_predict_may_2025()` 仅使用 `历史同月+同星期均值` 作为预测值，同一星期的每一天得到的预测值完全相同。真实世界中，即使是同一个星期二，不同周的销售额也必然存在波动。

**严重程度**：中等。虽然均值本身是合理的点估计，但完全相同的重复值在学术论文中会显得不可靠，且无法反映预测的不确定性。

**建议改进**：在均值基础上添加基于历史同星期标准差的随机波动（`pred = mean + noise × std`），或真正使用SARIMA/XGBoost做外推预测。

### 2.2 May 1st 是劳动节但被当作工作日预测

CSV中 `2025-05-01`（周四）被列为工作日并预测278人就餐。但2025年5月1日是国际劳动节，属于法定假日（且通常为5天长假的一部分）。代码中 `may_dates[may_dates.dayofweek < 5]` 仅排除了周六日，未排除法定假日。

**严重程度**：中等。预测表中混入了节假日，实际就餐人数在假期可能有显著不同。

### 2.3 午餐备菜方案 (`p3_meal_plan_detail.csv`)

**数据概览（5个工作日 × 50道菜 = 250条）**:

| 指标 | 值 | 评估 |
|------|-----|------|
| 求解状态 | 全天 Optimal | ✅ |
| 日均总备菜份数 | ~1,700份 | ✅ 与274人×5.5菜/人≈1,507接近 |
| 预期利润范围 | 471-546元/天 | ⚠️ 见下文分析 |
| 营养均衡度 | 0.93-0.94 | ✅ 良好 |

**问题1：预期利润偏低且与客单价矛盾**

日均274人 × 人均消费11.36元 = 日均3,113元收入。按45%成本率，毛利润约1,712元。但MILP模型输出预期利润仅471-546元，差距约3倍。

**根因**：MILP的收入项使用 `s_i = min(x_i, d_i)` 截断了收入上限（备菜量超过期望需求的部分不计入收入），而成本项计算的是全部备菜量的成本。此外，人均5.5道菜×274人=1,507份，但模型备菜约1,700份（含安全库存），多出的193份造成浪费成本。

**建议**：在目标函数中调整收入线和浪费惩罚的权重，或对"超出部分以折扣价销售"进行建模。

**问题2：分类错误导致荤菜误标为素菜**

CSV中出现以下不合理分类：
- `番茄炒蛋/红烧萝卜` → 被归类为"荤菜"（实际应为半荤半素）
- `黄豆猪脚` → 被归类为"素菜"（实际应为荤菜）
- `培红菜蘑菇毛豆肉丝` → 被归类为"荤菜"（合理，但含"菜""菇""豆"）
- `春卷` → 被归类为"荤菜"（实际为小吃/主食）

**根因**：关键词匹配中 `红烧` 关键词匹配了 `番茄炒蛋/红烧萝卜`，`猪脚`（可能不匹配`猪手`关键词）未命中，`春卷` 在荤菜关键词中定义了。

**严重程度**：中等。分类偏差影响多样性约束和套餐设计，但不根本影响优化结果的方向性。

**问题3：米饭单价仅0.21元**

备菜方案中米饭362份×0.21元。实际自助量贩餐厅的米饭通常定价2元左右一份。0.21元很可能对应"每克单价"而非"每份单价"——附件2的`unit_price`字段可能本身就是 `元/克` 而非 `元/份`。

**严重程度**：高。如果所有菜品的价格单位不统一（有的按份、有的按克），则价格约束、利润计算、套餐定价全部受影响。

### 2.4 套餐方案 (problem4输出)

前次测试输出的10元套餐存在**重复菜品**（白饭出现2次），这是贪心搜索的漏洞——`used_names`去重集合在跨类别选择时未能完全防止重复。

## 三、程序逻辑不足之处

### 3.1 数据层面

| 问题 | 文件 | 描述 |
|------|------|------|
| 价格单位不明确 | `data_loader.py` | `unit_price` 可能是元/克而非元/份，代码未做单位归一化，导致米饭仅0.21元 |
| 营养特征分类有误 | `data_loader.py:263-280` | `_nutritional_feature_classify` 将 `weight` 按 serving weight 处理并缩放到100g，但附件2中的weight值可能已是每份重量。缩放逻辑可能导致营养值误算 |
| 附件2覆盖率低的影响未量化 | `data_loader.py` | 附件2仅覆盖18.3%订单，但代码在构建购物篮和菜品偏好时100%依赖附件2。未讨论这18%子集与全量订单是否存在系统性偏差 |
| 停业日填充有歧义 | `problem2_prediction.py:141-145` | 补全日期范围时，将缺失的营业日数据填0并标记为is_closed。但236天中有365-236=129天数据缺失——这些是真的停业日，还是数据采集遗漏？ |

### 3.2 预测模型层面

| 问题 | 文件 | 描述 |
|------|------|------|
| 外推预测未使用训练模型 | `problem2_prediction.py:642-804` | `_predict_may_2025` 没有调用SARIMA或XGBoost进行真正的预测，而是回退到简单的历史同月同星期均值。花大量篇幅训练了3种模型却在外推时全部弃用 |
| 预测无变化性 | `problem2_prediction.py:690-713` | 同一星期几的所有日期预测值完全相同，无法反映周间波动。应在均值基础上添加噪声或置信区间 |
| 偏差修正因子恒为1.0 | `problem2_prediction.py:715-751` | XGBoost对最近60天的拟合均值与真实均值几乎相同，导致 `bias=1.0`，偏差修正完全没有发挥作用 |
| SARIMA拟合可能失败但不影响后续 | `problem2_prediction.py:317-324` | SARIMA失败时静默填充NaN，ensemble权重为0。代码逻辑正确，但未在结果中标记SARIMA的失败情况 |
| 未对节假日建模 | `problem2_prediction.py` | 春节、五一、国庆等重大节假日对餐饮业影响显著，但特征中无节假日标记 |
| MAPE值可能失真 | `utils.py:35-77` | `mape_score` 过滤 `y_true=0` 样本（停业日），但停业日本身就是需要预测的标志 |

### 3.3 优化模型层面

| 问题 | 文件 | 描述 |
|------|------|------|
| MILP收入截断导致利润低估 | `problem3_optimization.py:326-334` | 使用 `s_i ≤ x_i` 和 `s_i ≤ d_i` 线性化 min 函数，但 d_i 的估算（total_demand × pop_i）依赖附件2的偏好度，可能存在偏差 |
| 备菜方案跨日独立 | `problem3_optimization.py:517-600` | 5个工作日各自独立求解，未考虑食材采购的批量折扣、库存跨日结转等实际约束 |
| 成本率一刀切45% | `data_loader.py:366` | 所有菜品使用统一45%成本率。实际上荤菜成本率可能60%+（肉类贵），素菜成本率可能25%（蔬菜便宜） |
| 未使用问题2的预测结果 | `problem3_optimization.py:602-626` | `_get_predicted_diners` 使用同星期历史均值，而非调用问题2的预测模型。两个"预测"模块彼此独立，不符合"预测驱动优化"的设计意图 |
| 偏好度归一化可能失真 | `problem3_optimization.py:148-156` | `lunch_popularity = lunch_count / sum(lunch_count)`，即假设每个菜品被选择的概率正比于历史频率。但历史频率受备菜量影响（供应充足的菜自然卖得多），存在因果倒置 |

### 3.4 套餐设计层面

| 问题 | 文件 | 描述 |
|------|------|------|
| 局部优化的random.choice未固定seed | `problem4_combos.py:452` | `op = random.choice(['replace', 'add', 'remove'])` 未传入 `random_state`，可能导致结果在每次运行时略有差异 |
| 贪心搜索可能产生重复菜品 | `problem4_combos.py:375-419` | 因 `used_names` 集合在跨结构类别时才检查，偶尔出现重复（如前期测试中10元套餐出现2次白饭） |
| 共现关联矩阵构建效率低 | `problem4_combos.py:191-199` | 双重循环O(n²)计算共现，n=212时约22,366对。采样3000个订单降低了复杂度但可能遗漏稀有搭配 |

## 五、可改进方向（按优先级排序）

### Priority 1（影响结论可靠性）

1. **外推预测应使用训练模型**：修改 `_predict_may_2025` 使用 SARIMA 的 `forecast(steps=...)` 方法和 XGBoost 的递归预测，而非仅依赖历史均值
2. **价格单位校准**：检查附件2的 `unit_price` 字段实际含义，对所有菜品做单位归一化
3. **预测添加不确定性**：输出预测的置信区间（±1σ），而非单点估计

### Priority 2（提升模型严谨性）

4. **节假日日历集成**：使用 `chinese_calendar` 或 `holidays` 库标记法定假日和调休工作日
5. **MILP收入模型修正**：调研实际自助量贩的计费方式（按重量？按份？），相应调整收入函数
6. **成本率差异化**：荤菜/素菜/主食使用不同的成本率，而非统一45%
7. **问题2与问题3串联**：问题3直接读取问题2的预测DataFrame，实现真正的预测-优化联动

### Priority 3（优化细节与鲁棒性）

8. **套餐搜索添加去重逻辑**：在 `_greedy_search` 的结果验证阶段检查并删除重复菜品
9. **SARIMA失败统计**：在 `results` 中添加各目标的SARIMA拟合成功率
10. **菜品分类准确率提升**：扩充 `CATEGORY_KEYWORDS` 词典（如添加'猪脚'、'春卷→小吃'），或使用基于BERT的中文菜名分类
11. **中文字体回退方案**：在matplotlib配置中添加字体查找逻辑，自动检测可用中文字体
